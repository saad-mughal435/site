hft-orderbook · benchmark

Decode-to-book latency.

This measures one thing only: how long it takes to decode a single NASDAQ ITCH 5.0 message and apply it to the order book. No network IO, no parsing setup, no allocation up front - just decode + book.apply per message, timed with steady_clock. The session is a deterministic synthetic ITCH 5.0 feed (seed=1), about 192k messages.

These numbers come from GitHub Actions ubuntu-latest - shared, virtualized hardware. Treat them as relative figures (which message type is cheaper, where the tail sits), not absolute wall-clock you would see on a tuned box. That is the same caveat the repo README carries.

Latency by message type

Bar length is proportional to the selected percentile, in nanoseconds. Lower is faster.

max values are tail outliers - a single scheduling hiccup or cache miss on shared CI hardware. The percentiles (p50/p95/p99) are the signal; the max is noise you should expect.

Reproduce it

The benchmark target ships in the repo and writes the exact JSON this page renders. On any machine with CMake and a C++17 compiler:

Shell

cmake -S . -B build -DCMAKE_BUILD_TYPE=Release -DHFTOB_BUILD_BENCH=ON && cmake --build build -j && ./build/latency_json 200000 latency.json
